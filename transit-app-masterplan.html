Interview Guide — 8 Participant Script
Click each question to reveal what we're testing underneath it.
1
"Walk me through your last bus journey. What went wrong, if anything?"
+
Testing: Real pain points unprompted. We're looking for: uncertainty language ("I wasn't sure if…"), timing failures ("I missed it by…"), workaround behaviours (calling friends, checking Google Maps instead). Listen for spontaneous app comparisons.
2
"Show me how you currently figure out when your bus is coming."
+
Testing: Current tool ecosystem (competitor analysis). Note: do they use a transit app, Google Maps, a paper timetable, or nothing? How many taps does it take? Where do they drop off?
3
"If you could wave a magic wand and fix one thing about public transport, what would it be?"
+
Testing: Aspirational needs vs. baseline needs. This surfaces whether users want better information (our product space) or infrastructural problems we can't solve. Redirects to "what if you knew the answer instantly?" if they say "more buses".
4
"How many minutes before a bus arrives do you head to the stop — and why that number?"
+
Testing: Buffer time = anxiety proxy. Users who add 10+ minutes are the highest-anxiety segment, our primary early adopters. This also calibrates what notification timing means to users ("leaving in 5 min" vs "leaving in 10 min").
5
"Show me your phone — would you open an app or look at a notification to get a bus time?"
+
Testing: Entry point preference. Validates whether push notification is the killer feature vs. in-app search. Also surfaces glanceability requirements — widget support may be needed earlier than planned.
6
"You're at the stop, the app says bus arrives in 8 minutes — but what if it doesn't?"
+
Testing: Trust and accuracy expectations. Determines how we communicate uncertainty (confidence intervals vs. hard ETAs). Users who say "I'd never trust an app again" are churn risks if accuracy is below threshold — sets our ETA accuracy SLA.
7
"Would you pay £1.99/month for guaranteed accurate bus times? Why or why not?"
+
Testing: Price sensitivity and willingness to pay. If >60% say yes, premium freemium model is viable. Expect most to say "it should be free" — the growth hacker strategy accounts for this with a data monetisation model instead.
8
"If this app worked perfectly, who else do you know who'd want it?"
+
Testing: Viral coefficient seed. Identifies natural sharing moments (sharing a stop link, recommending to family). Used to design the referral mechanic — "Share stop ETA with friends" feature originates from this question.
Key Hypotheses to Validate
  • H1
    Primary frustration is uncertainty, not frequency — Users are bothered more by "not knowing when" than "buses are too infrequent"
  • H2
    Push notifications are the killer feature — Users want to be told, not to look it up
  • H3
    Trust is binary — One wrong ETA erodes trust for weeks; accuracy matters more than features
  • H4
    Map is secondary to ETAs — Most users want numbers, not visual bus locations

🔌 API Endpoints
GET /api/v1/stops/nearby Stops within radius
Query: ?lat=51.5074&lng=-0.1278&radius=500 Response: { "data": [{ "id": "uuid", "name": "High St / Market", "lat": 51.5074, "lng": -0.1278, "distance_m": 142, "next_arrivals": [...] }], "meta": { "total": 5, "radius_m": 500 } }
⚡ Cached in Redis for 30s. Geospatial index on (lat, lng). P95 <80ms.
GET /api/v1/stops/:id/arrivals Next buses at stop
Response: { "data": [{ "route": "42A", "destination": "City Centre", "eta_seconds": 183, "eta_display": "3 min", "vehicle_id": "uuid", "confidence": 0.92, "is_realtime": true }], "meta": { "stop_id": "...", "generated_at": "ISO8601" } }
⚡ Real-time from GTFS-RT. Falls back to schedule if feed down. Confidence score drives UI colour (green/amber/red).
GET /api/v1/vehicles/live All active bus positions
?bbox=51.48,-0.15,51.52,-0.10 // bounding box filter ?route=42A // optional route filter
⚡ Reads directly from Redis ZSET geospatial index. Refreshed every 10s from GTFS-RT feed. 1000 vehicles max per response.
POST /api/v1/alerts Create departure alert
Body: { "stop_id": "uuid", "route_id": "uuid", "notify_minutes_before": 5, "days_of_week": [1,2,3,4,5] }
🔒 Requires auth. Stores alert, schedules background job. Max 10 alerts per user. Returns 201 with alert ID.
POST /api/v1/journeys/plan Plan A→B journey
{ "origin": { "lat": 51.50, "lng": -0.13 }, "destination": { "lat": 51.52, "lng": -0.09 }, "depart_at": "ISO8601 | null (= now)" }
Uses modified Dijkstra on route graph. Returns up to 3 options sorted by duration. Includes walk segments, fare estimates.

🔄 Viral Loop Design
The core viral mechanic: Share a Stop Link.

When a user checks a stop's live arrivals, they can tap "Share" to send a pre-formatted link (transittrack.app/stop/high-st?route=42A) to a friend via WhatsApp, iMessage, or Instagram Stories.

The shared link opens a beautiful web preview of live arrivals at that stop — with a download prompt. This is the most natural sharing moment ("running late, check if the bus is coming").

K-factor target: If each user shares once per week and converts 0.3 installs, K = 0.3 per sharing event. Compounding over 4 weeks = ~1.2× base growth. Aim to improve this to K > 1.0 with incentivised referrals.

🔄 Prototype → Production Transition
When prototype validates (DAY 4 test passes):
• Supabase → self-hosted PostgreSQL on RDS (production schema from backend architect plan)
• PostgREST → custom Express.js API with auth, rate limiting, full error handling
• Redis Cloud free → ElastiCache cluster with failover
• Expo Go → EAS standalone build with code signing
• Mock analytics → production Amplitude with full funnel tracking

Estimated transition time: 3–4 days with existing code as the UI foundation.
Rapid Prototyper Rule: Never spend more than 4 hours on something that hasn't been validated. The map and ETA cards are the product — everything else (onboarding, settings, profile) ships as the bare minimum needed to test the core hypothesis.
